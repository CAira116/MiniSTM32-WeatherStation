🔧 W1D6：手写 GPIO 函数库

2026-07-10 · MiniLib-F103 第一块积木 · 从寄存器到自己的函数

📌 今天你做了什么？
以前点灯要写 5 行寄存器操作，每次都翻手册。今天你把它们装进了 4 个函数 —— gpio_init、gpio_write、gpio_read、gpio_toggle， 以后一行搞定，这辈子不用再手写 GPIOA->CRL &= ~(0xF << 20)。

0. 先搞清楚：我们到底在干什么？

0.1 GPIO 是什么？

STM32 芯片外面有 一排金属引脚（pin），每个引脚可以当输入（读外面电压） 也可以当输出（自己输出 3.3V 或 0V）。控制引脚的硬件模块叫 GPIO （General Purpose Input/Output，通用输入输出）。

你的工作就是：通过写寄存器，告诉 GPIO 模块每个引脚干什么活、什么时候干。

0.2 寄存器又是什么？

芯片内部有一堆"配置开关"，每个开关管一件事。寄存器 = 一组 32 位（bit）的开关组。 写 1 = 开，写 0 = 关。CPU 通过对特定内存地址读写来操作这些开关。

把寄存器想象成遥控器：
· CRL / CRH = 功能选择键（输入？输出？多大电流？）
· BSRR = 瞬时开关（按一下就生效，松手不变回去）
· IDR = 状态指示灯（只读，告诉你引脚现在几伏）
· ODR = 晾衣架状态（翻转用：看看现在是高还是低）

0.3 为什么要把寄存器"包起来"？

裸写寄存器（你 W1D3 的代码）：

// 点灯，5 行
RCC->APB2ENR |= 1 << 2;           // 1. 开 GPIOA 时钟
GPIOA->CRL &= ~(0xF << 20);      // 2. 清 PA5 配置位
GPIOA->CRL |= (0x3 << 20);       // 3. 设 PA5 为推挽输出
GPIOA->BSRR = 1 << 5;            // 4. PA5 拉高，灯亮
// ... 想灭灯？又一行
GPIOA->BSRR = (1 << (5 + 16));   // 5. PA5 拉低，灯灭

三周后回来看这段代码：0x3 是什么？20 从哪来的？为什么是 5+16？全忘了，从头翻手册。

封装成函数后：

gpio_init(GPIOA, 5, OUT_PP);     // 一句话：PA5 设为推挽输出
gpio_write(GPIOA, 5, 1);         // PA5 高 —— 灯亮
gpio_write(GPIOA, 5, 0);         // PA5 低 —— 灯灭
gpio_toggle(GPIOA, 5);           // 翻转 —— 亮变灭、灭变亮

不用翻手册，不用背移位，看一眼就知道在干嘛。这就是封装的意义。

1. 工程结构：三个文件各管各的

文件角色（人话）放什么
gpio.h 📋 目录 / 菜单 宏定义（给数字起名）+ 函数声明（告诉编译器"有这些函数"）
gpio.c 🔧 工具箱本体 四个函数的完整实现代码 —— 每个大括号里的内容
main.c 🎬 主程序 / 使用者 调用 gpio_init、gpio_toggle 等来干具体的事（比如闪灯）
🔑 核心规则（记住这一条就不会再报错）：
.h 负责"宣告" → .c 负责"实现" → main.c 负责"调用"
每个 .c 文件第一行必须 #include "自己的.h"，否则编译器不认识你的类型。

2. 逐个拆解：gpio.h 里每一行是什么意思

gpio.h 
#ifndef __GPIO_H          // "如果还没有定义 __GPIO_H"
#define __GPIO_H          // "那就定义它"
                          // 两人配合 = 防止同一个文件被包含两次
#include "stm32f10x.h"    // 拉进 ST 官方的寄存器定义（GPIOA、RCC、uint8_t...）

// ── 模式宏：给四位二进制数起个能看懂的名字 ──
#define IN_FLOAT    0x04   // 浮空输入   → CNF=01 MODE=00 → 0100 = 0x04
#define IN_PU       0x08   // 上拉输入   → CNF=10 MODE=00 → 1000 = 0x08
#define OUT_PP      0x03   // 推挽输出   → CNF=00 MODE=11 → 0011 = 0x03
#define OUT_OD      0x07   // 开漏输出   → CNF=01 MODE=11 → 0111 = 0x07
#define AF_PP       0x0B   // 复用推挽   → CNF=10 MODE=11 → 1011 = 0x0B

// ── 函数声明：告诉编译器"这些函数存在，定义在别的文件里" ──
void    gpio_init(GPIO_TypeDef *port, uint8_t pin, uint8_t mode);
void    gpio_write(GPIO_TypeDef *port, uint8_t pin, uint8_t value);
uint8_t gpio_read(GPIO_TypeDef *port, uint8_t pin);
void    gpio_toggle(GPIO_TypeDef *port, uint8_t pin);

#endif   // 结束 #ifndef 的区域

2.1 #define 到底干了什么？

#define OUT_PP 0x03 就是个纯文本替换器，在编译之前执行：

你写的代码：    gpio_init(GPIOA, 5, OUT_PP);
                ↓ 预处理器把 OUT_PP 换成 0x03
编译器看到的：  gpio_init(GPIOA, 5, 0x03);

为什么是 0x03？打开 STM32 参考手册 GPIO 章节，找到 CRL 寄存器描述中这张表：

CNF1CNF0MODE1MODE0二进制十六进制含义
001100110x03推挽输出，50MHz
011101110x07开漏输出，50MHz
010001000x04浮空输入
100010000x08上拉输入
101110110x0B复用推挽，50MHz
💡 培养写代码的能力：以后你碰到一个新寄存器，先翻手册找到类似的配置表，然后问自己——"我需要哪几个常用的组合？"列出来，给每个起个名字，写完宏定义。这比死记硬背十六进制强一百倍。

2.2 #ifndef ... #define ... #endif 三件套

// 设想：gpio.h 被 include 了两次
#include "gpio.h"    // ← 第一次：__GPIO_H 未定义 → 定义它，引入所有内容
#include "gpio.h"    // ← 第二次：__GPIO_H 已经有了 → 跳过全部内容

不写这个会怎样？如果 gpio.h 被多个 .c 包含，每个函数声明会出现两次，编译器报 "重复定义" 错误。

3. 逐个拆解：gpio.c 四个函数

gpio.c 
#include "gpio.h"    // ← 永远第一行！不写这行 = 全部类型名报错

3.1 gpio_init —— 初始化（告诉芯片这个脚干什么活）

void gpio_init(GPIO_TypeDef *port, uint8_t pin, uint8_t mode)
{
    // ═══ 阶段一：开时钟 ═══
    if (port == GPIOA)
        RCC->APB2ENR |= 1 << 2;       // GPIOA 在 APB2 总线 bit2
    else if (port == GPIOB)
        RCC->APB2ENR |= 1 << 3;       // GPIOB 在 APB2 总线 bit3
    else if (port == GPIOC)
        RCC->APB2ENR |= 1 << 4;       // GPIOC 在 APB2 总线 bit4

    // ═══ 阶段二：写配置 ═══
    uint8_t offset;                    // 该 pin 在寄存器里的起始位位置

    if (pin < 8) {                     // pin 0-7 → 用 CRL（低 8 个引脚）
        offset = pin * 4;              // 每个 pin 占 4 位 → 位置 = pin号 × 4
        port->CRL &= ~(0xF << offset); // ① 先清空这 4 位（写成 0000）
        port->CRL |= (mode << offset);  // ② 再写入新模式
    } else {                           // pin 8-15 → 用 CRH（高 8 个引脚）
        offset = (pin - 8) * 4;        // 例如 pin10 → (10-8)×4 = 8 → 从 bit8 开始
        port->CRH &= ~(0xF << offset);
        port->CRH |= (mode << offset);
    }
}

为什么 pin × 4？

每个 GPIO 引脚在配置寄存器里固定占 4 位：

CRL 寄存器（32位）： bit: 31-28 27-24 23-20 19-16 15-12 11-8 7-4 3-0 pin7 pin6 pin5 pin4 pin3 pin2 pin1 pin0 4位 4位 4位 4位 4位 4位 4位 4位 pin5 的配置在 bit20-bit23 → 起始偏移 = 5 × 4 = 20 ← 这就是 offset

清空这 4 位是怎么做到的？

port->CRL &= ~(0xF << offset);

// 逐步拆解，假设 offset = 20 (pin5)：

0xF                         = 0000 0000 0000 0000 0000 0000 0000 1111
0xF << 20                  = 0000 0000 0000 1111 0000 0000 0000 0000  （四个1对准bit20-23）
~(0xF << 20)               = 1111 1111 1111 0000 1111 1111 1111 1111  （取反，四个0对准目标位）

CRL &= 上面这个值           → 把 bit20-23 清零，其他位保持不变
🔑 &= ~ 是嵌入式最常用的"清零指定几位"咒语：
· 0xF → 你要清几位？（4 位 = 0xF，8 位 = 0xFF）
· << offset → 移到目标位置
· ~ → 取反，变成"目标位全 0，其余全 1"
· &= → 0 的地方清零，1 的地方不动

写入新模式：

port->CRL |= (mode << offset);

// 继续 pin5 的例子，mode = OUT_PP = 0x03 = 0011(binary)

0x03 << 20 = 0000 0000 0000 0011 0000 0000 0000 0000

CRL |= 上面             → bit20=1, bit21=1, bit22=0, bit23=0
                        → CNF0=1, CNF1=0, MODE0=1, MODE1=1
                        → 推挽输出 50MHz ✓

3.2 gpio_write —— 输出高/低电平

void gpio_write(GPIO_TypeDef *port, uint8_t pin, uint8_t value)
{
    if (value)                            // value 是 1（非零）→ 走这条
        port->BSRR = (1 << pin);           // BSRR 低 16 位写 1 = 拉高
    else                                  // value 是 0 → 走这条
        port->BSRR = (1 << (pin + 16));   // BSRR 高 16 位写 1 = 拉低
}

BSRR 寄存器的工作原理（这是今天最精妙的东西）

BSRR 是个 32 位寄存器，拆成两半： 高 16 位（bit 16-31）：复位区 → 写 1 到 bit(16+n) → 引脚 n 拉低 低 16 位（bit 0-15） ：置位区 → 写 1 到 bit n → 引脚 n 拉高 写 0 到任何位 → 什么都不发生（硬件自动忽略）

举例：要拉高 PA5 → port->BSRR = (1 << 5) → bit5 = 1，PA5 高。

举例：要拉低 PA5 → port->BSRR = (1 << 21) → bit21 = 1，PA5 低。（21 = 5 + 16）

为什么用 BSRR 而不是直接改 ODR？
ODR 需要"读 → 改 → 写"三步，如果中间被中断打断、别的代码改了同端口其他引脚，你的修改就丢了（这叫 read-modify-write 竞态）。BSRR 直接写目标位，硬件保证原子操作，不会被中断搅黄。

if (value) 怎么判断走哪条路？

gpio_write(GPIOA, 5, 1);   // value = 1  → 1 是非零 → if (1) = 真 → 走第一条路：拉高
gpio_write(GPIOA, 5, 0);   // value = 0  → 0 是零   → if (0) = 假 → 走第二条路：拉低

3.3 gpio_read —— 读引脚状态

uint8_t gpio_read(GPIO_TypeDef *port, uint8_t pin)
{
    if (port->IDR & (1 << pin))   // IDR 的 pin 位是 1 吗？
        return 1;                   // 是 → 返回 1（高电平）
    else
        return 0;                   // 不是 → 返回 0（低电平）
}

IDR（Input Data Register）是只读寄存器，直接反映外部引脚电压：>2V → 对应位=1；<0.8V → 对应位=0。

// 使用示例：
uint8_t btn = gpio_read(GPIOA, 0);   // btn = 引脚 PA0 的当前状态（0 或 1）
if (btn == 0) {                       // 等于 0 说明 PA0 被拉低了（按键按下）
    gpio_write(GPIOA, 5, 1);         // 亮灯
}
🤔 为什么不直接 return port->IDR & (1 << pin)？
因为 (1 << 5) 的值是 32，不是 1。外面调用的人只想知道"高还是低"（0 或 1），你还得让他自己判断 32 是高还是低？封装的意义就是替调用者处理好这些脏活，返回干净的结果。

3.4 gpio_toggle —— 翻转电平

void gpio_toggle(GPIO_TypeDef *port, uint8_t pin)
{
    if (port->ODR & (1 << pin))        // 现在是高？
        port->BSRR = (1 << (pin + 16));  // → 拉低
    else                                 // 现在是低？
        port->BSRR = (1 << pin);          // → 拉高
}

原理：先读 ODR 确认当前状态 → 反着写。高就写低，低就写高。

ODR（Output Data Register）记录每个引脚当前输出的是什么电平。跟 IDR 的区别：IDR 是引脚实际电压（外部可能被拉高/拉低），ODR 是芯片想输出的电压。

4. main.c —— 用你的函数点灯

main.c 
#include "gpio.h"        // 拉进 gpio_init / gpio_toggle 的声明

void SystemInit(void) {}    // 启动文件需要这个函数，空着就行

void delay(volatile uint32_t n)   // volatile = 禁止编译器把循环优化掉
{
    while (n--);
}

int main(void)
{
    gpio_init(GPIOC, 13, OUT_PP);    // PC13 设为推挽输出 —— 只一行！

    while (1) {
        gpio_toggle(GPIOC, 13);      // 翻转 —— 也只一行！
        delay(5000000);              // 延时让你肉眼看见闪
    }
}

5. 今天踩的坑：编译器把 delay 吃掉了

5.1 现象

灯不闪，一直亮。明明延时了 500000，肉眼应该看得见闪烁。

5.2 原因

编译器看到：

void delay(uint32_t n)   // ← 没有 volatile
{
    while (n--);          // 编译器：这个循环什么有用的都没干？删了。
}

编译器的优化策略是"等价变换"——既然 while 循环不改变任何东西，它等价于什么都不做，直接删掉。你的 500000 次循环被压缩成了 0 次。

5.3 修复

void delay(volatile uint32_t n)   // volatile 告诉编译器：n 随时可能变，别自作聪明
{
    while (n--);                    // 编译器：好吧，老老实实循环
}
⚠️ volatile 什么时候用？
· 延时循环变量 → 加 volatile，否则被优化掉
· 指向寄存器的指针 → 必须 volatile（硬件值会自己变，编译器不知道）
· 中断里修改的全局变量 → 加 volatile，否则编译器可能读到旧值
· 普通变量 → 不用加，让编译器优化
