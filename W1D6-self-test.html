第 1 题：引脚在寄存器里占几位？pin7 在 CRL 的哪几个 bit？

相关代码：

// gpio.c — gpio_init 中的偏移计算
uint8_t offset;

if (pin < 8) {                     // pin 0-7 → CRL
    offset = pin * 4;               // ← 为什么乘 4？
    port->CRL &= ~(0xF << offset);
    port->CRL |= (mode << offset);
} else {                            // pin 8-15 → CRH
    offset = (pin - 8) * 4;
    port->CRH &= ~(0xF << offset);
    port->CRH |= (mode << offset);
}

CRL 寄存器位布局：

bit  31-28 | 27-24 | 23-20 | 19-16 | 15-12 | 11-8  | 7-4   | 3-0
      pin7 | pin6  | pin5  | pin4  | pin3  | pin2  | pin1  | pin0

答案：每个引脚占 4 位。pin7 在 CRL 的 bit 31-28。

每个引脚 = CNF[1:0] + MODE[1:0] = 4 位。pin7 是 CRL 最后一个（bit31-28），pin0 是第一个（bit3-0）。由此得到：offset = pin × 4。

常见错误：以为 pin7 在 bit 7-4。记住——是按 pin 编号从低到高排，pin0 在最低 4 位，pin7 在最高 4 位。

第 2 题：0xF << offset 里的 0xF 为什么是四个 1？要清 3 位该用几？

相关代码：

// gpio.c — 清零目标 4 位的咒语
port->CRL &= ~(0xF << offset);   // 0xF = 1111(binary) = 4 个 1

逐步拆解（offset = 20）：

0xF                         = ... 0000 0000 0000 1111   ← 4个1
0xF << 20                  = ... 1111 0000 ...(20个0)   ← 4个1移到bit20-23
~(0xF << 20)               = ... 0000 1111 ...          ← 取反，bit20-23=0
CRL &= ~(0xF << 20)        = 只把bit20-23清零，其他不变

答案：0xF = 1111（二进制）= 4 个 1 = 代表 4 位。要清 3 位用 0x7（0111 = 3 个 1）。

要清几位十六进制二进制
1 位0x10001
2 位0x30011
3 位0x70111
4 位0xF1111
8 位0xFF1111 1111

&= ~(mask << offset) 是嵌入式最常用的位操作咒语，今天以后必须能背出来。

第 3 题：BSRR 拉高 PA3 写哪个 bit？拉低 PA3 写哪个 bit？

相关代码：

// gpio.c — gpio_write
void gpio_write(GPIO_TypeDef *port, uint8_t pin, uint8_t value)
{
    if (value)
        port->BSRR = (1 << pin);              // ← 拉高
    else
        port->BSRR = (1 << (pin + 16));        // ← 拉低
}

BSRR 寄存器结构：

高 16 位 (bit 16-31)：复位区 → 写 1 = 引脚拉低
低 16 位 (bit 0-15) ：置位区 → 写 1 = 引脚拉高
任何位写 0 → 不产生效果，硬件自动忽略

答案：拉高 PA3 → 写 bit 3（1 << 3）。拉低 PA3 → 写 bit 19（1 << (3 + 16)）。

同一个 pin 号，打在低半区 = 置高，打在高半区 = 置低。

常见错误：以为拉低是 BSRR = (0 << pin)——写 0 到 BSRR 什么都不发生！拉低的正确姿势是往高 16 位写 1。

第 4 题：#define OUT_PP 0x03 和 gpio_init(GPIOA,5,OUT_PP) 之间，预处理器做了什么？

相关代码：

// gpio.h — 模式宏定义
#define OUT_PP      0x03   // 推挽输出 50MHz
#define OUT_OD      0x07   // 开漏输出 50MHz
#define IN_PU       0x08   // 上拉输入
#define IN_FLOAT    0x04   // 浮空输入
// 你写的代码：
gpio_init(GPIOA, 5, OUT_PP);

// ▼ 预处理器处理后，编译器实际看到的：
gpio_init(GPIOA, 5, 0x03);

答案：预处理器在编译前执行纯文本替换，把 OUT_PP 直接替换成 0x03。

#define A B = "全文搜索 A，换成 B"。不检查类型、不计算、不理语义——就是最粗暴的查找替换。

所以写 #define OUT_PP 0x03 时：

  • OUT_PP 随便起名，但最好全大写（约定俗成）
  • 0x03 必须准确对应手册里的 CNF+MODE 组合
  • 宏后面不要加分号！#define OUT_PP 0x03; ← 分号也被替换进去，会炸
第 5 题：今天的 delay 不加 volatile 灯就不闪——原因一句话说清楚。

相关代码：

// ❌ 不闪的版本
void delay(uint32_t n)     // ← 没有 volatile
{
    while (n--);            // 编译器认为这个循环"什么都没干"，直接优化删除
}

// ✅ 能闪的版本
void delay(volatile uint32_t n)   // ← 加了 volatile
{
    while (n--);                   // 编译器：好吧，老老实实每次循环检查 n
}

现象回顾：

// main.c 调用
gpio_toggle(GPIOC, 13);
delay(5000000);              // 肉眼应该看到明显闪烁
gpio_toggle(GPIOC, 13);
delay(5000000);              // 但如果 delay 被优化掉 → 两次 toggle 几乎同时发生 → 看起来常亮

答案：编译器优化时认为空循环"无副作用"，直接删除，导致两次 toggle 几乎同时执行（微秒级），肉眼看不到闪烁。

volatile = "这个变量可能随时被外部改变（中断/硬件），每次用到它必须从内存重新读，不准缓存，不准删相关操作。"

场景加 volatile？
延时循环变量✅ 加
指向硬件寄存器的指针✅ 必须加
中断里修改的全局变量✅ 必须加
普通局部变量❌ 不加
